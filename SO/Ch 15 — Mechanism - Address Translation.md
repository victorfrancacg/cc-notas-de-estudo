# Ch 15 вҖ” Mechanism: Address Translation

> [!abstract] One-sentence summary
> The general technique for memory virtualization is **hardware-based address translation**: every load/store generated by the [[Process]] is rewritten by the [[MMU]] from a [[Virtual Address|virtual address]] to a physical one, in flight, transparently. This chapter introduces the simplest form: **[[Base and Bounds]]** (dynamic relocation) вҖ” two registers, two checks, almost free in hardware.

## Crux

> [!example] The Crux: How To Efficiently And Flexibly Virtualize Memory
> How can we build an *efficient* virtualization of memory? How do we provide the *flexibility* needed by applications? How do we maintain *control* over which memory locations an application can access? How do we do all of this *efficiently*?

## 15.1 Assumptions (To Be Relaxed Later)

The chapter starts with three deliberately unrealistic simplifications:

1. The user's address space is placed **contiguously** in physical memory.
2. The address space is **smaller than physical memory**.
3. Every address space has the **same size**.

Each of these will be relaxed in subsequent chapters ([[Segmentation|Ch 16]], [[Paging|Ch 18]], swapping in Ch 21).

## 15.2 The Idea: Hardware-Based Address Translation

> [!info] Interposition is the technique
> The hardware **interposes** on every memory reference. The CPU emits a virtual address; the [[MMU]] silently translates it to a physical address before the memory bus sees anything. The program is unaware translation is happening вҖ” that's the **transparency** goal of [[Memory Virtualization]].

The same interposition trick used in [[Limited Direct Execution]] (let the program run, but trap on certain events) returns here for memory: let every load/store pass through, but check it.

## 15.3 [[Dynamic Relocation]] = [[Base and Bounds]]

The simplest possible address-translation hardware:

```
physical_address = base + virtual_address
if (virtual_address >= bounds) RaiseException(out_of_bounds)
```

Two CPU registers per process:

- **base** вҖ” physical address where this process's address space starts.
- **bounds** вҖ” size (or top) of the address space.

The process *thinks* its address space is 0..16K. Every reference adds 32K and is checked against 16K. **That's it** вҖ” the entire mechanism.

> [!tip] The OS just sets the registers
> Translation is *all hardware*. The OS's only runtime job is updating `base`/`bounds` on a [[Context Switch]] (and creating/freeing process slots).

## 15.4 Hardware Support Needed

| Hardware feature | Why |
|---|---|
| Base & bounds registers | Per-process translation state |
| Privileged instructions to set them | Only the kernel can change them ([[User Mode vs Kernel Mode]]) |
| Out-of-bounds trap | Hardware raises a fault on illegal access |
| Privileged "switch to user mode" | Kernel returns to user code with translation enabled |

## 15.5 OS Support Needed

The OS gains three small new responsibilities:

1. **At process creation** вҖ” find a free slot of physical memory big enough for the process's address space; record base/bounds in its [[Process Control Block]].
2. **At process termination** вҖ” return the slot to the **free list**.
3. **At [[Context Switch]]** вҖ” save the outgoing process's base/bounds, load the incoming one's. (When the process is not running, the OS may even relocate it вҖ” copy the slot, update base.)

> [!example] The two timelines (Figs 15.5 + 15.6)
> OSTEP gives two beautiful timelines: **boot-time** (OS sets up trap table, timer, process table, free list) and **runtime** (OS allocates a process, hardware translates each access, timer fires, OS context-switches, an out-of-bounds trap kills a misbehaving process). They visualize how thin the OS layer is once the hardware is doing its job.

## 15.6 Limitation: [[Internal Fragmentation]]

Each process gets a **fixed-size slot** in physical memory. If the address space is 16KB but the process only uses 4KB of code+heap+stack with 12KB of unused middle, those 12KB sit *inside* the allocated slot вҖ” wasted. This is **internal fragmentation**, the principal weakness of base/bounds. Ch 16 (segmentation) attacks it.

## 15.7 Summary

> [!example] Crux answered
> The OS controls and enforces translation by setting **base/bounds** registers; the hardware does the actual translation on every memory reference. The result is a **transparent**, **efficient**, **protected** virtualization вҖ” but a wasteful one (internal fragmentation). The chapters ahead refine the mechanism.

## Aside: "Interposition Is Powerful"

> [!tip] Beyond memory
> Hardware interposes between virtual and physical addresses. The same idea вҖ” interpose on a well-defined interface to add new functionality without changing client code вҖ” appears throughout systems: virtual file systems, network stacks, hypervisors, eBPF. Memorize the pattern.
