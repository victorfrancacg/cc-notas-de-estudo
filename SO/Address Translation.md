# Address Translation

> [!abstract] Definition
> **Address translation** is the act — performed by the [[MMU]] on every memory reference — of converting a [[Virtual Address|virtual address]] generated by the [[Process]] into a **physical address** in DRAM. The OS sets up the translation tables; the hardware does the lookup. It is the central mechanism of [[Memory Virtualization|memory virtualization]].

## The Generic Picture

```
                 ┌──── MMU ────┐
   CPU pipeline ─→  translate ─→ memory bus
   issues VA      check perms    issues PA
                  ↑   ↑
            base/bounds, page tables, TLBs
            (set up by the OS)
```

Every load, store, and instruction fetch goes through this path. Translation must be **fast** (no programs would tolerate even microseconds per access) and **correct** (a wrong translation can cross the protection boundary).

## Mechanisms in OSTEP

| Mechanism | What it does | Strength | Weakness |
|---|---|---|---|
| [[Base and Bounds]] | one base + bounds for the whole address space | trivial hardware | [[Internal Fragmentation\|internal fragmentation]], no sharing |
| [[Segmentation]] | base + bounds per logical segment | sparse address spaces | external fragmentation |
| [[Paging]] | fixed-size pages, [[Page Table\|page table]] | no external frag | huge tables, slow walks |
| [[TLB]] | cache recent translations | makes paging fast | misses still costly |
| [[Multi-Level Page Table]] | paginate the page table itself | shrinks per-process tables | extra walk on miss |
| [[Inverted Page Table]] | one entry per physical frame | tiny on huge VAs | lookup needs hash |

Each is a refinement responding to a deficiency of the prior — the [[MOC - Memory Virtualization|Part II story]] is essentially this evolution.

## Why It's Always Hardware

A purely software translation would multiply every memory access by a function-call's worth of overhead. The CPU's MMU does it in parallel with cache lookup, hidden in the pipeline.

## Why the OS Still Matters

The OS owns the **translation tables** (per-process page tables, base/bounds values, segment registers). On every [[Context Switch]] it points the MMU at the next process's translations. On every page fault, it decides what to load and from where.

> [!tip] Mechanism vs Policy at the memory layer
> [[Mechanism vs Policy]] applies cleanly: the MMU is mechanism (fixed in silicon); the OS is policy (which physical frames to allocate, when to evict, how big page tables should be).

## Performance Vocabulary

- **TLB hit** — translation found in the MMU's cache. ~1 cycle.
- **TLB miss** — page-table walk. Tens to hundreds of cycles.
- **Page fault** — translation valid in the OS's records but the page isn't resident. Milliseconds (disk I/O).

The whole VM-system performance story is keeping these in roughly that ratio.
