# MOC — Memory Virtualization

> [!abstract] The Crux of Memory Virtualization
> **How can the OS give every running [[Process]] the illusion of its own large, private [[Address Space]] when only one physical memory exists?** By translating every [[Virtual Address|virtual address]] generated by user code into a physical one, with hardware ([[MMU]], [[TLB]]) doing the heavy lifting and the OS ([[Page Table|page tables]]) providing the policy. Part II walks the translation machinery from base/bounds to multi-level page tables.

## Core Concepts in Part II

### The Abstraction
- [[Address Space]] — the per-[[Process|process]] virtual view of memory.
- [[Virtual Address]] — what user programs generate.
- [[Physical Address]] — what DRAM actually uses.
- [[Memory Virtualization]] — the act of mapping virtual to physical.
- [[Code Segment]], [[Heap (Runtime)]], [[Stack (Runtime)]] — the canonical address-space regions.

### The Memory API
- [[malloc()]], [[free()]] — the user-visible heap interface.
- [[brk() and sbrk()]] — the lower-level kernel boundary.
- [[mmap()]] — anonymous and file-backed memory.
- Common pitfalls: [[Segmentation Fault]], [[Memory Leak]], [[Dangling Pointer]], [[Double Free]], [[Uninitialized Memory Read]].

### Mechanisms
- [[Base and Bounds]] — the minimal translation hardware.
- [[Dynamic Relocation]] — the technique base/bounds enables.
- [[MMU]] — Memory Management Unit; the hardware that translates.
- [[Segmentation]] — multiple base/bounds per logical region.
- [[Paging]] — fixed-size pages translated via a [[Page Table]].
- [[TLB]] — cache for recent translations.
- [[Multi-Level Page Table]] — paginate the page table itself.
- [[Inverted Page Table]] — one entry per physical frame, not per virtual page.

### Policies
- [[Free-Space Management]] — how the heap allocator picks blocks.
- [[Allocation Strategy]] — best-fit, worst-fit, first-fit, next-fit.
- [[Buddy Allocator]], [[Slab Allocator]] — kernel allocators.
- (Replacement policies arrive in Ch 22.)

### Goals (Recurring Theme)
- **Transparency** — programs unaware memory is virtualized.
- **Efficiency** — minimum time + space overhead.
- **Protection** — [[Isolation (OS)|isolation]] between processes and kernel.

## Classic Pitfalls (Preview)

> [!warning] External vs internal fragmentation
> [[Segmentation]] suffers external fragmentation; [[Paging]] suffers internal fragmentation. Each technique trades one for the other.

> [!warning] TLB misses dominate
> A [[TLB]] miss is *cheaper than a page fault* but **far more frequent** — VM-system performance is largely TLB performance.

> [!warning] Page tables can dwarf memory
> A 32-bit address space with 4KB pages → 4MB of page table *per process*. Multi-level and inverted tables exist to fix this.
